Microsoft Learning Components
Package Stores

In order to execute an e-learning package (i.e. create a StoredLearningSession onto it), Microsoft Learning Components needs to register the package. This process involves copying information from the package's manifest into various database tables for fast subsequent lookup. In the process, the package is assigned a PackageItemIdentifier, a 64-bit integer identifier that's unique within the database.

However, other than copying parts of the manifest, MLC doesn't store the package contents in its database. Instead, MLC stores a location string which is used to locate the package as needed. What's contained in the location string? It depends on where the application has decided to store its packages. The way the application tells MLC how to map a package identifier to package contents is by providing MLC with a package store. This section explains what a package store is, describes the package stores that MLC implements, and explains how an application can implement its own package store.

It's the application's job to determine the package store before creating a learning session(link). For that reason, it's simplest if all packages in a given LearningStore database use the same package store. However, your application is free to create a custom package store (i.e. a class based on PackageStore) that includes a prefix string indicating which package store to use. For that matter, you can create a custom package store that stores packages using any storage mechanism you like -- your package store will function essentially as a "driver", allowing MLC to locate your e-learning content.
